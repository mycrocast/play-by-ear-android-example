PlayByEarSDK

The PlayByEarSDK will enable you to easily integrate listening to live audio-broadcasts with low-latency, which were created via the mycrocast-studio website.

The following documentation will guide you through the setup and usage of the PlayByEarSDK for Android in more detail.

2. Initial Setup

The sdk itself is written in Kotlin and has a minimum requirement of Android 6.0 (API-Level 23).

Add the following maven-repository to your dependencyResolutionManagement in your settings.gradle file:

maven("https://mycrocast-maven-repo.s3.eu-central-1.amazonaws.com/android")

After this add the artifact of the PlayByEarSDK to your module dependencies, like this:

dependencies {
    implementation("de.mycrocast.sdk.play_by_ear:1.0.0")
}

Now you integrate the PlayByEarSDK into your application!


3. Integration of the SDK

The first thing you need to do to use the SDK in your application is to create an instance of the PlayByEarSDK as this class contains all classes you will need to integrate the functionality of the SDK.

Use the PlayByEarSDKBuilder to create an instance of the PlayByEarSDK. Hold the reference to this created instance for the whole lifetime of your application because you should only create it once and reuse it.

To create the PlayByEarSDKBuilder you need to create an implementation of the PlayByEarSDKCredentials interface as well as an implementation of the PlayByEarLogger and an instance of SharedPreferences. The last one is only used to store a unique identifier of the user of the PlayByEarSDK and therefore your application, as this allows us to identify reccurring users.


INFORMATION
The generation of the user identifier is completely randomized in the backend of PlayByEar and does not include any critical or sensitive information about the user, their mobile phone nor your application.


The PlayByEarSDKCredentials is needed to identify your application and to match the live audio-broadcasts of the streamers of your club, which started streaming via the mycrocast-studio website. Every club has a unique token, so you will only get informed about livestreams of your club.


INFORMATION
You will get your unique token from us after the registration. Alternative it can be found in the admin panel of your club in the mycrocast-studio website.


The PlayByEarLogger will give you additional information about the processes in the sdk itself like information, warnings and errors. While the information logs are just for understanding and tracking the process of the SDK, warnings may guide you to potential pitfalls and errors will occur only when an exception was thrown inside the SDK.


INFORMATION
Even if an exception was thrown the SDK itself will not crash. But it will help us a lot if you inform us if you encounter an error and how to reproduce it.


We would strongly advise to use a dependency injection library for the integration of the PlayByEarSDK, e.g. with Dagger-Hilt like in the example project. But it is also possible to setup the sdk in your custom application class or in a completely seperate singleton, if you want. But make sure you only create an instance once and reuse it.


4. Important interfaces

Before describing how to use the SDK and its interfaces in more detail, lets take a look to the core interfaces you will stumble on.

PlayByEarLivestream

The PlayByEarLivestream represents a single live audio broadcast which was started by a streamer in mycrocast-studio. It holds every information about a single audio-broadcast you will need, e.g a unique token, the title and language of the broadcast. (For more detail see the documentation)

PlayByEarLanguage

The PlayByEarLanguage holds information about the language of a PlayByEarLivestream. The selected language is represented in three different ways:

  • The english name of the language,

  • the code of the language, according to the ISO 639-1,

  • the native representation, how the language itself is called in the specific language (e.g. 'Deutsch' for German or 'Français' for French.


5. Usage of the SDK

After the setup and the configuration of the SDK in the first chapter, this one describes the intended way of using the SDK. For the following steps you need to use the interfaces given to you via the PlayByEarSDK.

The first thing you need to do is to use the PlayByEarConnection to connect to the backend of PlayByEar. If this succeeds you should load all currently active PlayByEarLivestreams via the PlayByEarLivestreamLoader. This will change the list of active PlayByEarLivestreams in the PlayByEarLivestreamContainer. If a user selects an active PlayByEarLivestream to play, you should start a Foreground Service to play the audio-broadcast of this livestream via the PlayByEarLivestreamPlayer, which can be created with the PlayByEarLivestreamPlayer.Factory.

Establishing a connection and observing changes

For establishing a connection as well as observing the connection state use PlayByEarConnection. At the start before the first connection attempt the initial value of currentState will be NEW. This is the perfect time to try establishing a connection to the PlayByEar backend via connect. You will need an active network connection for this to succeed. While trying to establishing a connection, the currentState changes to CONNECTING. Here you could show a loading animation, if you want. Depending on whether the connection attempt succeeds or fails the currentState changes either to CONNECTED for success or DISCONNECTED for failure.

If the connection attempt fails, you will get informed by the reason via the PlayByEarLogger. Of course you can try to establish a connection again either via connect or reconnect.

If the connection attempt succeeds, you can load all currently active PlayByEarLivestreams as described below.

If you established a connection once it is possible that this connection is lost after some time. This is exactle the case if the currentState changes from CONNECTED to DISCONNECTED. Here you need to implement a reconnect use case (again via connect or reconnect) according to your specifications. We would advise you to reconnect on demand (e.g. the user has a reconnect button) if no livestream is currently playing and if one is playing the application should try to reconnect in a timely manner (see the example application).


INFROMATION
While the SDK is not connected, the following requests will fail:

  • PlayByEarLivestreamLoader: load

  • PlayByEarLivestreamPlayer: play

If a request fails, the SDK will log it as an error in your provided PlayByEarLogger.


Loading and observing PlayByEarLivestream changes

After establishing a connection to the PlayByEar backend you can request all currently active live broadcasts via the PlayByEarLivestreamLoader. Therefore just call the function load. If this succeeds, meaning if it returns true, it will update the property online containing the list of currently active PlayByEarLivestream in the PlayByEarLivestreamContainer. In the beginning, before the first loading process succeeded, this property contains an empty list. If the process fails, returning false, no changes in the PlayByEarLivestreamContainer will be made.

If a PlayByEarLivestream was added (a new livestream was started), changed (streamer adjusted the livestream) or removed (streamer stopped streaming) the list of the online property will be adjusted automatically and you will get informed about changes if you collect this flow.


INFORMATION
If the connection to the PlayByEar backend was lost and after some time reconnected, it is necessary that you reload the list of currently active livestreams, as this could have changed in the time the client was disconnected.


To get updates of a specific PlayByEarLivestream you can collect the flow of the findByStreamerId(streamerId: Long) method of the PlayByEarLivestreamContainer, where the parameter streamerId is the identifier of the streamer of the corresponding PlayByEarLivestream. You should do this in your Foreground Service, which is playing the PlayByEarLivestream. This flow will contain the value null exactly when no PlayByEarLivestream with the given streamerId was found. If you collect the flow in the Foreground Service and the value is null, the streamer either stopped the live audio-broadcast of the PlayByEarLivestream or has problems with his internet connection (triggered by a timeout in the backend). Here you could stop the service immediately or wait for a specific time, because the streamer can reconnect to his live audio-broadcast. If the streamer reconnects, the value of the flow will be the reconnected PlayByEarLivestream, which has a different token than the previous one. This means you need to call 'play()' of the PlayByEarLivestreamPlayer again with the token of the new PlayByEarLivestream to continue the play.

Playing a PlayByEarLivestream

For playing a PlayByEarLivestream you need to create a PlayByEarLivestreamPlayer via the PlayByEarLivestreamPlayer.Factory.

After the creation of a PlayByEarLivestreamPlayer the initial value of current, alias the current PlayState is New. If you want to start playing a PlayByEarLivestream call the play method with the token of the PlayByEarLivestream. This will change current to Connecting and the player tries to establish a connection to the audio-broadcast of this PlayByEarLivestream. If this succeeds current will change to Playing, if it fails it will change to Failed.

If a connection was established previously (meaning current holds the value Playing) and the connection was lost (due to connectivity issues on client or streamer side), current changes to Disconnected and after some time it will change to Failed. Each of these PlayStates holds the token alias streamToken of the PlayByEarLivestream.

If you want to stop the playing of a PlayByEarLivestream invoke the method stop of the PlayByEarLivestreamPlayer you used to start the playing process. This will also stop an ongoing connecting process (current is still Connecting and not yet Playing). After stopping is done current will be Closed.


INFORMATION
First i would strongly advise to only use the PlayByEarLivestreamPlayer in a Foreground service, as his functionality was only tested in one and can be only guaranteed in one.

Also i would recommend to create a new PlayByEarLivestreamPlayer via the Factory for each Foreground service you start and not reuse old ones as this can lead to unwanted and unpredictable PlayState changes.

For an example how to use a Foreground service with the PlayByEarLivestreamPlayer, which also handles connection changes of client and streamer side, look into the LivestreamPlayService of the example application


6. How to test or How to create a broadcast?

Login

Visit the mycrocast-studio website and in the upper tab select "Streamer" and login with the credentials of a streamer of your club.

Start stream

Hit the stream start button and:

  • Select a genre

  • Select a language

  • Select Event

  • Enter a title

  • Select a microphone

  • Hit continue -> you are live a few moments later

Use audio file as content

Instead of your own voice you can use an audio file which will be broadcasted. To do so:

  • Drag and drop a file into the upper right corner of the website

  • Hit the play button on the entry that appeared there
